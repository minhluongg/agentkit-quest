The `git-manager` is a small, focused [subagent](/guides/skills-vs-commands-vs-subagents)
that handles the git ritual — stage, write a clean commit, push, open a PR — so your main
conversation isn't cluttered with the mechanics of version control.

## What it does

Stages changes, writes [Conventional Commits](/skills/git) (no AI-attribution trailers),
pushes, and manages branches and PRs. It's the delegated version of the
[`/ak:git` skill](/skills/git).

## Why it's a subagent

Git operations are procedural and self-contained — exactly the kind of task worth handing
off so you stay focused on the code, not the commit dance. It's a small agent doing one
thing reliably.

## It runs on Haiku

AgentKit routes it to **Haiku** — writing a commit message from a diff is a light task, so
it uses the cheap fast model. This is the model-routing pattern throughout the kit: match
the model to the job's difficulty.

## Build your own (free)

```md
---
name: git-manager
description: Stage, commit (conventional), and push changes. Delegate git operations here.
tools: Bash, Read
model: haiku
---

Write Conventional Commits from the staged diff: type(scope): subject under 72 chars.
Never add "Generated by" or AI co-author trailers. Confirm before pushing to a protected
branch.
```

Pair it with the [free commit-enforcement hook](/guides/claude-code-hooks-examples) and the
rules are enforced whether or not the agent remembers them.

## When it earns its place

- As the "ship" step in a [plan → build → review → ship](/guides/agentkit-first-project) loop.
- When you want commits and PRs handled without thinking about them.
