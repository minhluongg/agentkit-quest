Auth is the code you least want an AI to improvise. Session handling, OAuth flows, and
token storage have sharp edges where "looks fine" and "actually secure" diverge. The safe
path is to use a real auth library — [Better Auth](https://better-auth.com) is a good
TypeScript one — and have Claude Code wire it correctly rather than hand-roll it.

## The job

Add authentication — email/password, OAuth, 2FA, passkeys, sessions — that's actually
secure, not a homemade token scheme.

## How to do it without a kit

**1. Never let Claude hand-roll auth.** No custom JWT signing, no bespoke session logic.
Use a maintained library. Prompt:

> Set up authentication using Better Auth. Do not write custom crypto or token handling —
> use the library's primitives. Configure email/password and Google OAuth to start.

**2. Get the token storage right.** Sessions in httpOnly cookies, not localStorage
(localStorage is readable by any XSS). Have Claude explain *where* tokens live and why —
if it can't, don't ship it.

**3. Secrets from the environment.** OAuth client secrets, signing keys — env vars only. A
[free hook](/guides/claude-code-hooks-examples) blocks a secret from being written into a
file if you slip.

**4. Test the failure paths.** Expired tokens, revoked sessions, wrong password — these
are where auth bugs hide. [Write tests for them](/skills/test).

Better Auth's docs plus Claude Code cover email/password, OAuth, 2FA, and passkeys for
free.

## What the kit's skill adds

AgentKit's `better-auth` skill (`/ak:better-auth`) packages the setup patterns — OAuth
providers (Google, GitHub), 2FA/MFA, passkeys/WebAuthn, sessions, RBAC, rate limiting —
with scripts, so each auth feature is a guided add rather than a docs-diving exercise.

```bash
/ak:better-auth "add Google OAuth and 2FA"
```

*(Legacy prefix: `/ck:better-auth`.)*

## We have not run this one

Other skill pages here show what the skill actually printed against this repository. This one has
no transcript.

**This site has no authentication.** No accounts, no sessions, no cookies — a
[security audit](/skills/security) of the whole repo found it has *no identity to spoof at all*,
which is why five of the six STRIDE categories are structurally inapplicable to it. There was
nothing here to protect.

Scaffolding an auth flow purely so this page had a screenshot would have proved that
`better-auth` can produce a login form. It would not have told you the thing you want to know —
how it behaves against a real user model with real edge cases — and we would rather show the gap
than fill it with something that looks like an answer.

## When NOT to reach for it

- **Auth is a one-time setup and you have done it before.** The library's own docs are good. This
  packages the patterns; it does not replace understanding them, and you will be living with the
  result for years.
- **You need a hosted identity provider.** Better Auth is a library you run. If the answer is
  Clerk, Auth0 or WorkOS, this is the wrong tool and the skill will not tell you so.
- **You are debugging a session bug.** Setup patterns will not find it.
  [`/ak:debug`](/skills/debug) — prove the cause before you change the auth flow, because auth is
  the worst place to fix a symptom.

## Combining it

- **Adding auth to an app:** `/ak:plan` → `/ak:better-auth` for the setup →
  [`/ak:security`](/skills/security) for the threat model. Those are different questions: one asks
  "does this work", the other asks "who benefits if it doesn't".
- **Auth touches the schema:** [`/ak:databases`](/skills/databases) for the user and session
  tables — get them wrong and the migration is painful later.
- **Before it ships:** [`/ak:code-review`](/skills/code-review). Auth code is the code where a
  plausible-looking mistake is most expensive.

## Buy if / Don't buy if

- **Worth it if** you set up auth across projects and want the OAuth/2FA/passkey patterns
  ready-made.
- **Skip it if** it's one app — Better Auth's own docs plus Claude Code get you there free.
