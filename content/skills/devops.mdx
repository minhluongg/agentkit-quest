Infrastructure work is unforgiving of small mistakes — a wrong `kubectl` context, a
Dockerfile that busts its cache, a Terraform apply against the wrong environment. Claude
Code is genuinely useful here *if* you keep it deliberate. Here's the free approach and
what the skill adds.

## The job

Build and operate infra — containers, orchestration, cloud resources — with Claude driving
the standard tools, without a careless `apply` to production.

## How to do it without a kit

The CLIs are free; Claude runs them well. What matters is the guardrails:

**Docker:** ask Claude to write a Dockerfile that orders layers for cache efficiency
(dependencies before source) and uses a multi-stage build to keep the image small. It's
good at this once you ask for it explicitly.

**Kubernetes:** the danger is context. Have Claude *print the current context and namespace
before every mutating command* — `kubectl config current-context`. A [free hook](/guides/claude-code-hooks-examples)
can enforce confirmation before destructive commands.

**Cloudflare (Workers/R2/D1), GCP (Cloud Run/GKE):** `wrangler` and `gcloud` do the work;
Claude reads your config and runs the right commands.

The rule across all of it: **read before you write.** Have Claude show you the plan/diff
(`terraform plan`, `kubectl diff`) before applying. Infra mistakes are the expensive kind.

## What the kit's skill adds

AgentKit's `devops` skill (`/ak:devops`) covers Cloudflare (Workers, R2, D1), Docker, GCP
(Cloud Run, GKE), and Kubernetes (kubectl, Helm) with scripts and a security-audit angle,
so common infra flows are guided rather than assembled command-by-command.

```bash
/ak:devops docker "containerize this app"
/ak:devops kubernetes "deploy to GKE"
```

*(Legacy prefix: `/ck:devops`.)*

## We have not run this one

Other skill pages here show what the skill actually printed against this repository. This one has
no transcript.

**This site has no containers.** No Dockerfile, no Kubernetes manifests, no cloud infrastructure
of its own — it is a static Next.js build that a CDN serves. There was no infra to point the skill
at.

Writing a throwaway Dockerfile so this page had a terminal block would have demonstrated that the

out.

## When NOT to reach for it

- **You have one Dockerfile and it works.** Container tooling is where cargo-culting is most
  expensive: a config you did not write, do not understand, and cannot debug at 3am is worse than
  a simple one you do.
- **The cluster is already on fire.** This guides infra flows; it does not diagnose a failing
  deploy. Read the logs, then [`/ak:debug`](/skills/debug).
- **Your host does not need containers.** Vercel, Netlify and Cloudflare Pages take a git push.
  Reaching for Docker because it feels more serious is how a simple site becomes an infra project.

## Combining it

- **Containerising an app:** [`/ak:backend-development`](/skills/backend-development) for the app
  → `/ak:devops` for the container → [`/ak:security`](/skills/security), because a container image
  is a supply chain and it is the part people forget to audit.
- **Shipping it:** `/ak:devops` for the infra → [`/ak:deploy`](/skills/deploy) for the release.
- **CI:** the pipeline is where a test suite stops being decoration. If nothing runs your tests,
  you do not have tests — you have files.

## Buy if / Don't buy if

- **Worth it if** you work across Docker + K8s + a cloud regularly and want guided flows
  with guardrails.
- **Skip it if** you're on one platform — its CLI plus deliberate prompting is enough.
