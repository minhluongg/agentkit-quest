The most expensive Claude Code mistakes are architectural — a wrong approach taken three
files deep, which is cheap to fix in a plan and painful to unwind in code. Planning first
is the single highest-leverage habit, and Claude Code has built-in support for it.

## The job

Turn a vague task ("add notifications") into a concrete plan — the steps, the files it will
touch, the trade-offs — and review that plan *before* any code exists.

## How to do it without a kit

**1. Use plan mode.** Claude Code can explore and propose an approach without changing
anything until you approve. Start there for anything non-trivial. Prompt:

> Before writing any code, produce a plan: the approach, the files you'll create or modify,
> the trade-offs of your chosen approach vs the alternatives, and the risks. Don't edit
> anything until I approve.

**2. Make it write the plan to a file.** A plan on disk survives context compaction and
becomes the brief for the implementation step — the
[handoff that keeps a workflow coherent](/guides/claude-code-agent-orchestration).

**3. Attack the plan before you commit to it.** Ask Claude (or a fresh
[subagent](/guides/skills-vs-commands-vs-subagents)) to argue *against* the plan — find the
flaw, the missing case, the wrong assumption. A plan that survives an adversarial pass is
worth building; one that doesn't just saved you the rework.

That loop — plan, write it down, red-team it — is free and is most of the value.

## What the kit's skill adds

AgentKit's `plan` skill (`/ak:plan`) runs this as a structured workflow: research, codebase
scouting, a written phased plan, an optional adversarial red-team pass, and validation —
with modes for depth (`--hard`, `--deep`, `--parallel`) and TDD.

```bash
/ak:plan "add real-time notifications"
/ak:plan --hard "refactor the auth layer"
```

*(Legacy prefix: `/ck:plan`.)* This entire site was planned with this skill.

## What it actually printed

This one was not staged. **The work that produced the page you are reading was planned with
`/ak:plan`** — and then we turned the same skill hostile against its own output:

```bash
/ak:plan "Write real captured output onto the 20 published skill pages"
/ak:plan red-team plans/260713-1103-.../
```

The `red-team` subcommand spawns hostile reviewers, each with a different adversarial lens, and
**each must back every finding with a `file:line` citation from the actual codebase.** Findings
without evidence are rejected before adjudication.

**27 findings. 14 survived deduplication and verification. Four were Critical. The plan had to
be rewritten from scratch.**

It found a path from a credential to a public page:

````
[Critical] .env.local holds a live VERCEL_OIDC_TOKEN. The plan points
/ak:security-scan at "the repo" — and ak-security-scan/SKILL.md:46 runs secret
scanning FIRST, ALWAYS — then orders "paste. Do not summarise" and "never edit
what remains".

The full path: token → runs/security-scan.md → content/skills/security-scan.mdx
→ prerendered HTML → sitemap → Google → AI crawler caches.

No step in the plan looked for this.
````

It found that the plan's central safety claim was a sentence, not a mechanism:

````
[Critical] git branch --show-current → main. A merge to main is a live deploy.

  ak-git/SKILL.md:22   `cp` = "Stage files, create commits AND PUSH"
  ak-ship/SKILL.md:3   "merge main, test, review, commit, push, PR"
  ak-fix/SKILL.md:18   --auto is the DEFAULT

The plan says mutating skills "run on a scratch branch" — with no step to create
one, no check, no guard.
````

And it caught three things the plan had simply asserted:

````
Plan claimed                                  Verified reality
─────────────────────────────────────────────────────────────────────────────────
"Two-way guide links ✅ Holds"                4 of 20. Sixteen pages are orphans.
"/skills/plan cannibalises the guide"         Zero keyword overlap. The proposed
                                              "fix" would have discarded a real
                                              head term.
The 20 OWNER keywords (copied from the map)   10 of 20 differ from what the pages
                                              actually ship. Following the table
                                              would silently retarget 10 indexed
                                              URLs.
````

**One finding was rejected.** A reviewer declared the whole plan dead — *"the `ak` kit is not
installed […] so `/ak:` cannot resolve, and the real-output thesis is impossible."* It had checked
the project directory and missed the global install. Rejected on the evidence.

### What a reader should notice

**The plan we would have executed was dangerous, and we did not know it.** Every one of those
Criticals was invisible from inside the plan. They are things you find by *attacking* a plan, not
by reading it again more carefully.

**The `file:line` requirement is what makes it work.** A hostile reviewer with no evidence bar
produces plausible-sounding objections. This one had to cite `.gitignore:62` and the actual
`git check-ignore` output — so every finding could be checked, and one of them was checked and
thrown out.

**And it caught three of our own claims being false.** Confident prose in a planning document
reads exactly like verified fact, and nothing in the document distinguishes them.

## When NOT to reach for it

- **Small, reversible work.** Six phases of planning for a two-file change is the ceremony this
  skill's own `--fast` mode exists to skip.
- **You already know the shape.** Planning is for when the risk is that you are solving the
  wrong problem.
- **You will not act on the red team.** Running it and then talking yourself out of the findings
  is worse than not running it — you have paid for the counsel *and* built a record of ignoring
  it.

## Combining it

- **The full loop:** `/ak:plan` → `/ak:plan red-team` → `/ak:cook` →
  [`/ak:code-review`](/skills/code-review) → [`/ak:test`](/skills/test). The red team reviews the
  *plan*; the reviewer reviews the *code*. Both were needed on this site, and they found entirely
  different things.
- **Before planning, if the problem is fuzzy:** `/ak:brainstorm`. A plan is the wrong tool for
  deciding *what* to build — it will happily produce six confident phases for the wrong feature.
- **Scout first on an unfamiliar repo:** [`/ak:scout`](/skills/scout). A plan written without
  knowing where the code lives is a plan that will be rewritten.
- **When a phase turns up a bug:** [`/ak:debug`](/skills/debug) — prove it before you replan around
  it.

## Buy if / Don't buy if

- **Worth it if** you plan substantial features regularly and want research + phased plans +
  an adversarial pass with an evidence bar, as one command.
- **Skip it if** you plan occasionally — plan mode plus a "red-team this, and cite a file and
  line for every claim" prompt is the same discipline, free. **The evidence bar is the part that
  matters, and it costs nothing to demand.**
