Web testing spans three different jobs — does the UI work end-to-end (Playwright), do the
units behave (Vitest), does it hold under load (k6) — and Claude Code can wire up all
three. The trick is asking for tests that assert *behavior a user cares about*, not that a
button exists.

## The job

Cover a web app across e2e, unit, and load — with tests that would actually fail if the
app broke.

## How to do it without a kit

The tools are free and standard; Claude Code drives them well.

**End-to-end (Playwright):** ask Claude to write specs around *user journeys*, not
elements — "a user can sign up, log in, and see their dashboard," with assertions on the
outcome. Prompt it to run `npx playwright test` and fix failures against the real app, not
against its assumptions.

**Unit (Vitest):** [behavior-first tests](/skills/test) — the same discipline as any unit
testing.

**Load (k6):** describe the scenario ("100 users hitting /api/search for 60s") and have
Claude write the k6 script and interpret the p95/p99 output. The value is in reading the
results correctly, which Claude is good at once you give it the numbers.

A [free hook](/guides/claude-code-hooks-examples) can run the relevant test on save, so
you catch breakage as you edit rather than at the end.

## What it actually printed

We ran it against the **live production site**, not a local build — real Playwright and
Chromium, Core Web Vitals via a `PerformanceObserver` registered before any page script runs,
axe-core for WCAG A/AA, and a keyboard pass. Three runs per page, medians taken.

````
                          LCP        TTFB       CLS
Homepage                  396 ms     241 ms     0.0000
/skills/agent-browser     428 ms     232 ms     0.0000
Homepage, throttled       1500 ms    235 ms     0.0025   (4x CPU, Slow 4G)
Skill page, throttled     1484 ms    149 ms     0.0002

Targets: LCP < 2.5s · CLS < 0.1 · TTFB < 800ms
````

Accessibility came back at **0 violations across 22 rule-groups**. And here is the part worth
stealing — **a clean result is worth nothing if the scanner is silently no-op'ing, so it
falsified its own pass.** It injected known-broken markup into the live page and re-ran:

````
Injected: <img> with no alt · empty <button> · empty <a> · #bbb-on-white text

axe returned exactly 4:
  image-alt      [critical]
  button-name    [critical]
  link-name      [serious]
  color-contrast [serious]

The scanner detects. The clean result is real.
````

**And it caught a phantom bug before reporting it:**

````
Homepage run 1:  LCP 3600ms / TTFB 3480ms   ← looks like a failure

It isn't. curl shows the edge answering in 176-257ms across 5 requests, with
X-Vercel-Cache: HIT. And the THROTTLED run beat the unthrottled run 1.

The 3480ms was a cold Chromium process paying first-TLS and first-connection
cost — a cold BROWSER, not a cold server.

Had I run once and reported, I'd have filed a perf bug against a site that is fast.
````

### What a reader should notice

**The rigour is the deliverable, not the numbers.** Three habits separate this from a guess:
it took medians and discarded the warm-up; it cross-checked an alarming number with a second
instrument (`curl` said 200ms where the browser said 3.5s — one of them was measuring the
browser); and it falsified its own passing result, because *"0 violations"* and *"0 rules ran"*
produce identical output.

That last one is the habit worth copying. Most a11y reports do not do it, and most of them are
correct — but you cannot tell which ones from the report.

## When NOT to reach for it

- **Against localhost, if you want the truth about production.** A local `next start` has no
  CDN, no TLS handshake, no real network. It measures your app; it does not measure your site.
- **For a single number.** One run of anything is noise. If you are not taking medians, you are
  reporting the browser's mood.
- **Instead of a unit suite.** This tests the shipped page. It will not tell you *why* something
  broke — see [`/ak:test`](/skills/test) and [`/ak:debug`](/skills/debug).

## Combining it

- **After a rendering change:** [`/ak:web-frameworks`](/skills/web-frameworks) decides what is
  static; this measures whether that decision actually reached the user. The build manifest and
  the browser disagree more often than anyone expects.
- **Before trusting a perf "fix":** [`/ak:react-best-practices`](/skills/react-best-practices) will
  tell you a rule fires. This tells you whether it matters in a browser — on our own filter, the
  rule fired and the answer was 0.27ms.
- **Design and a11y together:** [`/ak:ui-ux-pro-max`](/skills/ui-ux-pro-max) reviews the code; this
  drives the real thing with a keyboard.
- **Turn a finding into a gate:** [`/ak:test`](/skills/test). A measurement is a snapshot; an
  assertion is a promise. Only one of them stops the regression.

## Buy if / Don't buy if

- **Worth it if** you want one skill spanning e2e/load/visual/a11y without configuring each
  framework — and you value the verification discipline above, which is most of what you are
  paying for.
- **Skip it if** you use one or two of these — Playwright + Vitest are free and Claude sets
  them up fine. The habits are free too; they are written above.
