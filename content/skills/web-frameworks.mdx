Next.js's App Router gives you several rendering strategies — Server Components, SSR, ISR,
static — and the whole game is picking the right one per route. Claude Code knows the
mechanics; the value you add is telling it the *constraints* so it chooses correctly.

## The job

Build Next.js features that use the right rendering strategy per route, with the caching
and data-fetching that fits — not everything jammed into a client component.

## How to do it without a kit

**1. Default to the server; justify every `'use client'`.** The most common Next.js
mistake is client-rendering things that should be Server Components. Prompt:

> Build this as Server Components by default. Only add `'use client'` where there's genuine
> interactivity, and tell me why each one needs it. Fetch data on the server where you can.

**2. Pick the rendering strategy from the data's freshness needs:**
- Static — content that rarely changes.
- ISR (`revalidate`) — changes occasionally; serve cached, refresh in the background.
- SSR — must be fresh per request.
- Client — depends on the user/browser at runtime.

Make Claude state the choice per route and the reason.

**3. Get caching right** — it's where App Router surprises people. Have Claude be explicit
about what's cached and how it's invalidated.

**4. Monorepos:** Turborepo's value is the task cache and shared packages; ask Claude to
set up the pipeline and workspace boundaries deliberately.
