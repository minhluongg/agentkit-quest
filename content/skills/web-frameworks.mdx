Next.js's App Router gives you several rendering strategies — Server Components, SSR, ISR,
static — and the whole game is picking the right one per route. Claude Code knows the
mechanics; the value you add is telling it the *constraints* so it chooses correctly.

## The job

Build Next.js features that use the right rendering strategy per route, with the caching
and data-fetching that fits — not everything jammed into a client component.

## How to do it without a kit

**1. Default to the server; justify every `'use client'`.** The most common Next.js
mistake is client-rendering things that should be Server Components. Prompt:

> Build this as Server Components by default. Only add `'use client'` where there's genuine
> interactivity, and tell me why each one needs it. Fetch data on the server where you can.

**2. Pick the rendering strategy from the data's freshness needs:**
- Static — content that rarely changes.
- ISR (`revalidate`) — changes occasionally; serve cached, refresh in the background.
- SSR — must be fresh per request.
- Client — depends on the user/browser at runtime.

Make Claude state the choice per route and the reason.

**3. Get caching right** — it's where App Router surprises people. Have Claude be explicit
about what's cached and how it's invalidated.

**4. Monorepos:** Turborepo's value is the task cache and shared packages; ask Claude to
set up the pipeline and workspace boundaries deliberately.

## What the kit's skill adds

## What it actually printed

### The input

```bash
/ak:web-frameworks src/
```

Target: this site — Next.js 16.2.10, App Router, React 19, static-first.

### It caught that the skill's own references are out of date

Before trusting the checklist, it checked the checklist:

````
Important caveat about the skill: its references are Next 13/14-era.

  references/nextjs-app-router.md:39      teaches the SYNCHRONOUS `params: { slug }` form
  references/nextjs-data-fetching.md:816  same
  references/nextjs-optimization.md:1300  teaches experimental.ppr

On Next 16 the repo is right and the skill is stale.
I used the skill for WHAT TO CHECK, not for WHAT IS TRUE.
````

**That paragraph is the most useful thing in the report.** A checklist written against Next 14
will confidently tell you to "fix" correct Next 16 code — and a model's priors about Next.js
are usually a year old too.

### It audited the build output, not the source's intentions

````
.next/prerender-manifest.json: 131 routes prerendered
/og is the ONLY app route not prerendered.

So: 100% static, plus one dynamic-by-necessity image endpoint. For a content site
whose data all lives in the repo, that is the correct architecture, not an anti-pattern.
````

### The real finding: a font subset nobody on this site can read

````
src/lib/fonts.ts:6,13 — subsets: ['latin', 'vietnamese']
src/lib/fonts.ts:3-4  — comment claims "It costs nothing today."

That comment is false, and I verified it against the built HTML.

.next/server/app/en.html emits exactly 4 <link rel="preload" as="font">:
  IBM Plex Sans,  latin
  JetBrains Mono, latin
  IBM Plex Sans,  unicode-range: U+1EA0-1EF9, U+20AB   ← Vietnamese
  JetBrains Mono,                                      ← Vietnamese

`rel="preload" as="font"` fetches at HIGH priority, UNCONDITIONALLY. `unicode-range`
gates lazy @font-face activation — it does NOT gate an explicit preload.

So half the eagerly-fetched font bytes on every page are glyphs this English-only
site can never render, competing with LCP resources.
````

Invisible in the source, whose comment asserts the opposite. Undeniable in the emitted HTML.

### And then it got one thing confidently wrong

It declared `/og` — the site's only dynamic route — **correct** to export no cache config,
reasoning from the library's source:

````
ImageResponse sets cache-control: public, immutable, max-age=31536000 in production
  (node_modules/next/dist/compiled/@vercel/og/index.node.js:21490)
Adding `dynamic` or `revalidate` to /og would range from useless to breaking.
````

The line it quotes is real. **The conclusion is false**, and we only know because we ran it:

````
$ npm run build && NODE_ENV=production npx next start
$ curl -sI localhost:3000/og?title=test | grep cache-control
cache-control: no-cache, no-store
````

The library's production branch never fires in a built app. `no-store` tells every CDN to keep
nothing — so every distinct `?title=` was a fresh function invocation rasterizing a PNG, billed
per request. A [security run](/skills/security) had flagged exactly this; the framework run read
the source and talked itself out of it.

**Both audits were half right, and neither had run the endpoint.**

## When NOT to reach for it

- **You already know App Router.** The server-first prompt above keeps Claude honest for free,
  and you will catch a stale reference faster than the skill will.
- **Your app is fully static.** Most of the Next.js audit surface — ISR windows, `use cache`,
  PPR, streaming boundaries, request-time cache headers — simply evaporates when every route
  is prerendered. The honest finding is usually that the headline features are *correctly
  declined*, not missed.
- **You want it to tell you what's true about the framework.** Use it for *what to check*.
  Verify *what is true* against the version you actually have installed.

## Combining it

- **Rendering strategy first, then perf:** `/ak:web-frameworks` decides static vs dynamic;
  [`/ak:react-best-practices`](/skills/react-best-practices) optimizes within that choice. Get
  the order backwards and you memoize your way around an architecture problem.
- **Anything it flags as a defect:** [`/ak:debug`](/skills/debug) to reproduce before you
  change it. Ours needed exactly that.
- **The one dynamic endpoint:** [`/ak:security`](/skills/security) — on a static site, the
  route that still executes *is* the attack surface.

## Buy if / Don't buy if

- **Worth it if** you build Next.js apps often and want the RSC/caching/monorepo patterns
  as a repeatable guide.
- **Skip it if** you know App Router — the server-first prompt above keeps Claude honest,
  free.

**Audit the build output, not the source's intentions.** That is this run's own best advice,
and the one place it failed to take it is the one place it was wrong.

The real risk in a static-first Next app is not "what should we cache" but "what could silently
opt a page out of static rendering" — and those bugs are invisible in the source and obvious in
`.next/`. This codebase has scars from exactly that: a `useSearchParams()` that once served
crawlers an empty hub page, and a proxy matcher that once 404'd every social card on the site.
