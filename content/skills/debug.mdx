The fastest way to debug slowly is to let Claude jump straight to a fix. It patches the
symptom, the bug moves, and you're two changes deeper with no idea what's actually wrong.
Effective debugging — with or without a kit — is a discipline: reproduce, prove the cause,
*then* fix.

## The job

Turn "it's broken" into a proven root cause and a fix that addresses it — not a guess that
happens to quiet the symptom.

## How to do it without a kit

The free method is a prompt that forbids the shortcut. Tell Claude, explicitly:

> Before proposing any fix: (1) reproduce this failure with a minimal test or script,
> (2) trace it to a specific root cause and show me the evidence, (3) only then suggest a
> change. Do not edit code until the cause is proven.

Then work the layers with it:

1. **Reproduce.** A failing test or a script that triggers it every time. If you can't
   reproduce it, you can't know you fixed it.
2. **Isolate.** Bisect — comment out, add logging, narrow the input — until the failure
   is localized to a few lines.
3. **Explain.** Have Claude state *why* it fails, in one sentence, citing the evidence.
   If it can't, it hasn't found the cause yet.
4. **Fix, then re-run the reproduction.** Green means done. Still red means the cause was
   wrong — back to step 2.

That discipline is 90% of debugging value, and it's free. The
[reproduce-before-fix workflow](/guides/claude-code-workflow-patterns) generalizes it.

## What the kit's skill adds

AgentKit's `debug` skill (`/ak:debug`) builds this loop in — root-cause-first by default,
with support for the harder cases the free prompt handles clumsily: call-stack tracing,
multi-layer validation, log analysis, CI/CD failure diagnosis, and database diagnostics.

```bash
/ak:debug "login returns 500 intermittently under load"
```

*(Legacy prefix: `/ck:debug`.)*

## What it actually printed

### The input

A real defect in this site, handed over **unproven** — an audit had spotted it by reading
the code:

> `guides/[...slug]/page.tsx:85` renders `{page.data.readingTime && (<span>…</span>)}`, and
> `readingTime` is `z.number().optional()` with no `.positive()`. So `readingTime: 0` should
> pass validation, `0 && …` should evaluate to `0`, and React should render a bare **"0"**.

```bash
/ak:debug "a guide with readingTime: 0 renders a stray 0 into the page metadata row"
```

### The output — it refused to stop at the code read

````
The mechanism is as the audit describes — but I did not stop at reasoning.
I reproduced it.

  1. Added `readingTime: 0` to content/guides/install-agentkit.mdx frontmatter
  2. npm run build  →  build PASSES, no zod error (confirms 0 slips through the schema)
  3. grep the real prerendered HTML
````

The proof, from the shipped HTML:

````
.next/server/app/en/guides/install-agentkit.html

...text-muted-foreground">beginner</span>0<span class="flex items-center gap-1.5"><svg ...
                                        ^
                                        the bare "0", in real output
````

A lone `0` between the closing tag of the difficulty badge and the opening tag of the
"Updated" block. The RSC payload carries the same `0` at the matching position in the tree —
so it came from *that* JSX branch, not by coincidence from other content. Then it reverted
the frontmatter and confirmed a clean tree.

### And then it found the thing that actually mattered

````
grep readingTime content/guides/*.mdx   →   0 results across all 14 guides.

So today, in production, page.data.readingTime is ALWAYS undefined.
`undefined && (...)` renders nothing. The entire "Clock icon + N min read" block
is 100% dead code.

The bug is real and fires the moment anyone sets readingTime: 0 — but its impact
today is zero, because the feature was never wired to real content.
````

## When NOT to reach for it

- **You already have a reliable reproduction.** If you can trigger it on demand, you are
  past the step this skill exists to enforce. Read the stack trace.
- **Reproducing is genuinely expensive.** This bug took one build. A race condition under
  production load, or a leak that needs a day of traffic, does not. The discipline is still
  right, but "reproduce it" is a cheap instruction to give and sometimes a costly one to
  follow — and the skill will not make it cheaper.
- **You want the fix, not the cause.** It will not give you one. That is the point.

## Combining it

- **The full loop:** [`/ak:scout`](/skills/scout) to find where it lives → `/ak:debug` to
  prove why → `/ak:fix` → [`/ak:test`](/skills/test) to lock the reproduction in as a
  regression guard.
- **When a review finds a bug:** [`/ak:code-review`](/skills/code-review) reports; `/ak:debug`
  proves. A reported bug is a hypothesis until someone reproduces it.
- **Turn the reproduction into a test.** The failing test *is* the reproduction — write it
  before the fix, and the bug can never come back unnoticed.

## Buy if / Don't buy if

- **Worth it if** you debug gnarly, multi-layer issues often and want the reproduce-first
  discipline enforced rather than remembered.
- **Skip it if** the prompt above is enough — for most bugs, "prove the cause first" typed
  into a bare Claude Code session gets you there.

**The lesson from our run is not about the skill; it is about the discipline.** The code read
gave the right answer and the wrong conclusion. *"The `&&` guard is wrong for `0`"* is true,
and on its own it produces a one-line fix — `.positive()` on the schema — that closes the bug
and misses the point entirely.

Building it exposed the real finding: **nobody uses this field.** Not one of fourteen guides.
The honest question was never "how do we make the guard safe" but "why is there a reading-time
display that nothing populates?" — and the quick fix would have buried it.

Reproducing cost a build. It bought a finding reasoning could not reach, because the reasoning
was about the code and the finding was about the *content*.
