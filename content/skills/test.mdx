AI-generated tests have a failure mode: they assert that the code does exactly what it
currently does. Every test passes, coverage looks great, and not one of them would catch
a bug — because they were written *from* the implementation, not from the intended
behavior. Here's how to get tests worth having.

## The job

Produce tests that would fail if the behavior broke — including the edge cases and error
paths, not just the happy path.

## How to do it without a kit

Two moves make Claude's tests actually useful:

**1. Test behavior, not implementation.** Prompt for the *contract*, not a mirror:

> Write tests for what this function should GUARANTEE, from its signature and docstring —
> not from its current implementation. Cover: the happy path, boundary values, empty and
> null inputs, and every error it can throw. For each test, the name should state the
> behavior being verified.

**2. Prove they can fail.** A test that can't fail is theater. Ask Claude to break the
implementation deliberately and confirm the relevant test goes red — then revert. If no
test catches the break, the tests are worthless and you've learned it cheaply.

For a bug, invert it: [write the failing test first](/guides/claude-code-workflow-patterns),
then fix until green. The test *is* the reproduction.

## What it actually printed

### The honest starting point

**This site had no tests.** No `test` script, no vitest, no jest. `playwright` sat in the
dependencies, imported by nothing.

That is worth stating plainly, because it exposes what the skill does and does not do:
**`/ak:test` runs a suite. It does not conjure one out of an empty repo.** So the run became
writing the first test — and the judgement about *what to assert* stayed with a human.

We picked the invariant the whole site rests on: **a reference page is indexable only once
someone has written real content for it.** Get that wrong and ~90 auto-generated pages ship,
which Google treats as scaled content abuse — and the penalty lands on the whole domain, not
just the thin pages that earned it.

Nothing is hardcoded. The expected sets are derived from the catalog and from
`content/skills/`, so the day the kit ships skill 92 the test still describes reality. A
literal count would go quietly wrong instead.

```ts
// tests/noindex-gate.spec.ts
const publishedSlugs = skills.map((s) => s.slug).filter((slug) => written.has(slug));
const stubSlugs = skills.map((s) => s.slug).filter((slug) => !written.has(slug));

test('the catalog splits into published pages and stubs', () => {
  // If either side is empty the tests below would pass vacuously, asserting nothing.
  expect(publishedSlugs.length).toBeGreaterThan(0);
  expect(stubSlugs.length).toBeGreaterThan(0);
});
```

### Then we broke it on purpose

**93 green proves nothing until you have watched it go red.** This is the step everyone
skips, and it is the only one that tells you whether the tests are real.

We added `noindex: true` to one published page's frontmatter — a plausible one-word mistake:

````
  ✘  4 › a page with hand-written content is indexable › /skills/code-review
  ✘ 93 › the sitemap lists every published page and no stub

  Error: expect(locator).toHaveAttribute(expected) failed
  Expected pattern: /(?<!no)index/
  Received string:  "noindex, follow"

  Error: expect(received).toContain(expected)
  Expected substring: "/skills/code-review<"
````

Two failures, on the two surfaces that matter: the crawler's robots directive, and the
sitemap that invited it. Reverted; green again.

Writing 93 assertions that all pass first time should make you suspicious, not pleased. Ours
did pass first time. We only know they are real because we broke the thing they watch.

## When NOT to reach for it

- **You have no tests yet.** The skill runs a suite; it does not decide what is worth
  asserting. That first decision is the whole game, and it is yours.
- **You want coverage numbers to look good.** Coverage measures which lines ran, not which
  bugs would be caught. A suite written from the implementation scores beautifully and
  guards nothing.
- **The thing you want to check is a build-time invariant.** A build assertion is cheaper and
  runs on every push. We use both: `verify-build.ts` checks the build output; these tests
  check what a crawler actually *receives* over HTTP. Different claims.

## Combining it

- **After implementing:** `/ak:cook` → [`/ak:code-review`](/skills/code-review) → `/ak:test`
- **For a bug:** [`/ak:debug`](/skills/debug) to prove the cause → write the failing test →
  `/ak:fix` → `/ak:test` until green. The test is the reproduction.
- **For the browser specifically:** [`/ak:web-testing`](/skills/web-testing) is the
  Playwright-shaped sibling — visual regression, a11y, Core Web Vitals.

## Buy if / Don't buy if

- **Worth it if** you want one command that runs and reports across unit/integration/e2e/
  visual, repeatedly.
- **Skip it if** you mostly need well-written tests — the behavior-first prompt above is
  the part that matters, and it's free.

The honest summary of our run: **the suite is worth more than the skill invocation that
produced it.** Everything valuable here — choosing the invariant, deriving the sets instead
of hardcoding them, the guard against vacuous passes, breaking it deliberately — is judgement,
not automation. The skill runs what you point it at. Point it at something worth running.
