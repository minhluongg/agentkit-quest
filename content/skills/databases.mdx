Claude Code is good at SQL and schema design — but it will happily generate a schema that
works for the query you mentioned and falls apart on the three you didn't. Good database
work is about modeling the *access patterns*, and that's a prompting discipline.

## The job

Design a schema (relational or document) and write queries that fit how the data is
actually read and written — with the right indexes and no N+1 traps.

## How to do it without a kit

**1. Give it the access patterns, not just the entities.** A schema designed from "users
have orders" differs from one designed for "show a user's last 10 orders by status." Tell
Claude *how the data will be queried*:

> Here are the entities and the top read/write patterns: [list them]. Design a Postgres
> schema optimized for these queries, with the indexes each one needs, and flag any query
> that would N+1.

**2. Ask for the indexes explicitly** — and *why* each exists. An index without a query
behind it is dead weight; a query without an index is a table scan. Have Claude tie each
index to the pattern it serves.

**3. For MongoDB**, the question is embed vs reference — driven by whether the sub-data is
read with the parent and how it grows. Make Claude justify the choice, not default to one.

**4. Review the query plan.** `EXPLAIN ANALYZE` (Postgres) turns opinion into evidence.
Have Claude read the plan and confirm indexes are used.

## When NOT to reach for it

- **You know SQL.** A skill will not out-think you on a query you can already write. Its value is
  in the parts that are tedious and easy to get subtly wrong — migrations, aggregation pipelines,
  index choices.
- **The schema decision is a product decision.** Multi-tenancy, soft deletes, event sourcing —
  these are business calls with database consequences, not database calls. Decide first, then ask.
- **You have a slow query and no measurements.** Get `EXPLAIN ANALYZE` output first.
  [`/ak:debug`](/skills/debug) exists for the same reason: a fix proposed without a reproduction
  is a guess.

## Combining it

- **Designing a feature:** `/ak:plan` for the shape → `/ak:databases` for the schema →
  [`/ak:backend-development`](/skills/backend-development) for the API over it.
- **A migration that must not lose data:** `/ak:databases` to write it →
  [`/ak:code-review`](/skills/code-review) on the migration itself → [`/ak:test`](/skills/test)
  against a copy. A migration is the one change you cannot casually revert.

## Buy if / Don't buy if

- **Worth it if** you do frequent schema/migration/query work and want it structured across
  Postgres and Mongo.
- **Skip it if** you design a schema occasionally — the access-pattern prompt above is the
  core skill, free.
