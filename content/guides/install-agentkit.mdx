The AgentKit CLI is `ak`. It installs as a **single native binary** — no Node.js, no Bun,
no npm package. If a guide tells you to `npm install -g claudekit-cli`, it is out of date:
that package is the deprecated ClaudeKit-era tool.

<Callout type="warn" title="Older guides are wrong here">
The rename from ClaudeKit to AgentKit also changed how it installs. The old method
(`npm install -g claudekit-cli`, then `ck …`) is deprecated. Use the native `ak` binary
below.
</Callout>

## Prerequisites

- **Claude Code** installed and working ([setup guide](https://code.claude.com)).
- That's it — `ak` is a standalone binary, so you do **not** need Node or a package manager.

## Install the CLI

<Steps>
  <Step>
    ### macOS / Linux

    ```bash
    curl -fsSL https://agentkit.best/install.sh | sh
    ```
  </Step>

  <Step>
    ### Windows (PowerShell)

    ```powershell
    irm https://agentkit.best/install.ps1 | iex
    ```
  </Step>

  <Step>
    ### Verify it worked

    ```bash
    ak --version
    ak doctor
    ```

    `ak doctor` checks your environment and reports anything missing.
  </Step>
</Steps>

<Callout type="tip" title="Prefer not to pipe a script into your shell?">
`curl … | sh` runs whatever the URL serves, unreviewed — a habit worth being deliberate
about. If you'd rather inspect first, download the script, read it, then run it:

```bash
curl -fsSL https://agentkit.best/install.sh -o install.sh
less install.sh   # read it
sh install.sh
```

(This is exactly the kind of thing one of our [free hooks](/guides/claude-code-hooks-examples)
blocks by default — worth setting up if you let Claude Code run install commands.)
</Callout>

## What the installer actually does

We ran this on Windows 11 on 2026-07-13. The real output:

```
[i] AgentKit signed installer
[i]   verifier: 2.2.0-beta.8 (windows/amd64)
[i]   channel:  stable
[i] downloading pinned bootstrap verifier
[OK] bootstrap verifier pin verified
[OK] verified release archive written to ...\verified-release.zip
[i] extracting authenticated release
[OK] installed ak.exe -> C:\Users\Admin\bin\ak.exe
WARNING: C:\Users\Admin\bin was added to your user PATH. Open a NEW terminal for it to take effect.
[i] ak 2.1.0
[OK] done. Run 'ak --help' to get started.
```

Two things worth knowing that the one-liner does not tell you:

- **You must open a new terminal.** The installer edits your user PATH; your current shell
  does not see `ak` until you restart it. Until then, call it by full path.
- **No admin rights are needed.** It installs to your user directory, not `Program Files`.

The installer is genuinely careful, which is worth saying because most `curl | sh` scripts
are not: it pins the SHA-256 and byte size of a bootstrap verifier, and that verifier checks
an Ed25519-signed release chain before a single byte of the CLI is trusted. It also **refuses
to run** if you try to skip checksum verification.

## The three steps the one-liner leaves out

<Steps>
  <Step>
    ### Log in — kits are licence-gated

    ```bash
    ak login --email you@example.com          # emails you an OTP
    ak login --license-key ak_license_...     # or activate with your licence key
    ak login --api-key ak_live_... --no-interactive
    ```

    Without this, `ak kit init` cannot fetch a paid kit at all. Verify with `ak whoami`,
    which lists the kits your account is licensed for.

    <Callout type="warn" title="The OTP path needs a real terminal">
    `ak login --email` prints `Enter OTP code:` and waits on stdin. Run it in an actual
    terminal — piped through a script or a non-interactive runner it submits an empty code
    and the server answers `400 Bad Request`, which looks like a rejected account but is not.
    Use `--license-key` or `--api-key` with `--no-interactive` when scripting.
    </Callout>
  </Step>

  <Step>
    ### Configure your provider key

    ```bash
    ak setup
    ```

    `doctor` reports `[X] API keys ... missing key(s): anthropic_api_key` until you do.
    `ak setup` is also what claims an existing `~/.claude` directory for AgentKit — see the
    warning below before you run it on a machine that already has ClaudeKit.
  </Step>

  <Step>
    ### Install the kit

    ```bash
    ak kit init engineer --target claude-code --global
    ```

    Drop `--global` to install into the current project only. The kit lands in
    `~/.claude/plugins/ak-engineer/` — check with `ak kit install-path engineer --json`
    before you commit to it.
  </Step>
</Steps>

<Callout type="warn" title="Stable ak cannot install the kit from the remote registry (as of 2026-07-13)">
Install the stable channel, then ask for the remote kit, and you get:

```
remote registry manifest requires newer ak: kit engineer requires
ak >= 2.2.0-beta.14 (current 2.1.0); upgrade ak first
```

The stable installer ships **2.1.0**; the registry's engineer kit wants **2.2.0-beta.14**.
The kits bundled inside the binary install fine — it is the `--remote` path that is blocked.
If you hit this, you need the beta channel: set `AK_CHANNEL=beta` before running the
installer, or run `ak self-update`.
</Callout>

<Callout type="warn" title="A global install takes over an existing ClaudeKit setup">
`ak kit init --global` performs a pre-install snapshot **and a ClaudeKit takeover** — this
is what the `--no-backup` flag exists to skip ("leaves CK content active", per its own help
text). If you have a working `ck` install you care about, know that going in.

To inspect a kit without touching your environment at all, build it to a scratch directory:

```bash
ak kit init engineer --target claude-code --build-only --out ./kit-inspect
```

That is exactly how we counted the kit for this site: 91 skills, 16 agents, 1,443 files
written, and nothing in `~/.claude` touched.
</Callout>

## You already have `ck` installed — what happens?

Nothing breaks. On our machine `ck` (claudekit-cli 4.5.0) still worked normally after
installing `ak`, and both binaries coexist:

```bash
$ ck --version
CLI Version: 4.5.0
Local Kit Version: engineer@v2.20.0

$ ak --version
ak 2.1.0
```

But `ak doctor` flags it as a **`ck` shim collision**, because AgentKit also wants to own a
`ck` shim for backwards compatibility and yours resolves to the old npm package. The fix is
to remove or rename the legacy `claudekit-cli`, or to put AgentKit's shim earlier on PATH.
Inspect what you actually have with `Get-Command ck -All` on Windows, or `which -a ck`
elsewhere.

## Start using it

Once the kit is installed, its skills are available in Claude Code under the `/ak:` prefix:

```bash
/ak:plan "add authentication to the app"
/ak:cook "implement the plan"
/ak:code-review
```

`/ak:plan` scopes and plans the work; `/ak:cook` implements it; `/ak:code-review` reviews
it. See the [skills catalog](/skills) for the full set, and the guide to
[skills vs commands vs subagents](/guides/skills-vs-commands-vs-subagents) if the
distinction is new to you.

## Coming from ClaudeKit?

If you already had ClaudeKit installed, the short version: the binary is `ak` not `ck`,
and commands use `/ak:` not `/ck:`. Full details in
[ClaudeKit is now AgentKit](/guides/claudekit-to-agentkit).
