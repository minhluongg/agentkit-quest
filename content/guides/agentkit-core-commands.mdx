AgentKit ships **91 skills**, but a handful carry the daily workflow. If you learn these, you
have the loop: plan it, build it, check it, ship it. Each is invoked with the `/ak:` prefix
in Claude Code (the old `/ck:` still works as a legacy alias).

<Callout type="info" title="The flags below are read from the installed kit, not invented">
Every argument list on this page comes from the `argument-hint` field in that skill's own
`SKILL.md`, in the Engineer kit (v0.2.0) we installed on 2026-07-13. Most of these flags are
not documented anywhere else — and several of them change what the command *does*, not just
how loudly it talks.
</Callout>

## The flags nobody tells you about

This is the part worth the read. These commands are not one-shot verbs; they have modes.

| Command | Arguments it actually accepts |
|---|---|
| `/ak:plan` | `[task] [--fast \| --hard \| --deep \| --parallel \| --two] [--tdd] [--no-tasks] [--html] [--github] [--wiki]` — or a subcommand: `archive`, `red-team`, `validate` |
| `/ak:cook` | `[task \| plan-path] [--interactive \| --fast \| --parallel \| --auto \| --no-test] [--tdd]` |
| `/ak:fix` | `[issue] [--auto \| --review \| --quick \| --parallel]` |
| `/ak:debug` | `[error or issue description]` |
| `/ak:test` | `[context]` — or `ui [url]` for browser testing |
| `/ak:docs` | `init \| update \| summarize` |
| `/ak:watzup` | *(no arguments)* |

Three that are worth knowing immediately:

- **`/ak:plan --two`** generates *two competing approaches* and makes you pick. For a design
  fork you cannot resolve by arguing with yourself, this is the whole feature.
- **`/ak:plan red-team`** turns the planner hostile against a plan it already wrote. Cheap
  insurance on anything expensive.
- **`/ak:cook --auto`** removes the human approval gates. Fast, and exactly as dangerous as it
  sounds — the gates are the point on anything you have not scoped tightly.

## The daily loop

| Command | What it does | When |
|---|---|---|
| [`/ak:plan`](/skills/plan) | Research + a written, phased plan before any code | Anything non-trivial |
| `/ak:cook` | Implement — runs plan → build → test → review as one workflow | To build the thing |
| [`/ak:code-review`](/skills/code-review) | Review a diff, PR, commit, or codebase for real bugs | Before merging |
| [`/ak:debug`](/skills/debug) | Root-cause-first debugging | When something's broken |
| [`/ak:test`](/skills/test) | Run and report across unit/integration/e2e | To validate |
| `/ak:fix` | Targeted fix with root-cause routing | For a known bug or failure |
| [`/ak:docs`](/skills/docs) | Init / update / summarize project docs | To keep docs in sync |

## The typical flow

<Steps>
  <Step>
    ### Plan

    ```bash
    /ak:plan "add email notifications"
    ```
    Produces a plan on disk — approach, files, trade-offs — to review before building.
    ([Why planning first pays off.](/guides/claude-code-workflow-patterns))
  </Step>

  <Step>
    ### Cook (implement)

    ```bash
    /ak:cook path/to/plan.md
    ```
    Executes the plan — and because `cook` runs testing and review as part of its workflow,
    you're not manually chaining steps.
  </Step>

  <Step>
    ### Review and ship

    ```bash
    /ak:code-review --pending
    ```
    A final check on the diff before it goes anywhere.
  </Step>
</Steps>

## Every one of these has a free method

None of these commands do anything Claude Code can't be prompted to do — they package a
discipline so you invoke it instead of remembering it. Each skill page here documents the
free way to do the same job:

- [Planning before you build](/skills/plan)
- [Debugging root-cause-first](/skills/debug)
- [Reviewing code well](/skills/code-review)
- [Generating real tests](/skills/test)

That's the honest framing of what a kit buys: not capability you're locked out of, but a
pre-built, consistent workflow. Whether that's worth paying for is
[its own question](/guides/agentkit-vs-free-alternatives).
