A Claude Code skill is one file: `SKILL.md`. Frontmatter at the top tells Claude when
the skill is relevant; the body is the instructions Claude follows once it loads. That's
the whole model. This page shows the exact anatomy, a complete example you can copy, and
the handful of mistakes that stop a skill from firing.

Skills are a [free, first-party feature](/guides/skills-vs-commands-vs-subagents) — you
do not need any paid kit to write them.

## Where the file goes

```text
.claude/
  skills/
    my-skill/
      SKILL.md          ← required
      reference.md      ← optional supporting files
      script.py         ← optional scripts the skill can run
```

The directory name (`my-skill`) becomes the invocation: `/my-skill`. A skill can also be
a single file at `.claude/commands/my-skill.md` if it needs nothing but instructions —
that is the [simpler "command" form](/guides/skills-vs-commands-vs-subagents).

## Anatomy of a SKILL.md

```md
---
name: commit-helper
description: Write a Conventional Commit message from the staged diff. Use when the user asks to commit, or says "write a commit message".
---

You write Conventional Commit messages. When invoked:

1. Run `git diff --staged` to see what changed.
2. Pick the type: feat, fix, docs, refactor, test, chore.
3. Write a subject line under 72 characters: `type(scope): description`.
4. If the change is non-trivial, add a body explaining *why*, not *what*.
5. Never add AI attribution or "Generated by" trailers.

Show the message and ask for confirmation before running `git commit`.
```

Two parts, and both matter:

- **`description`** is not documentation — it is the **trigger**. Claude reads it to
  decide whether to load the skill automatically. Write it as "what this does + when to
  use it," with the words a user would actually say. A vague description means the skill
  never fires on its own.
- **The body** is the instruction set. It loads into context *only when the skill is
  used*, so it can be long without costing tokens on every turn. Write it as clear,
  numbered steps — Claude follows procedures far more reliably than prose.

## Control who invokes it

By default Claude may load a skill automatically when the description matches, and you can
always invoke it explicitly with `/name`. You can tighten this in frontmatter:

| You want | Frontmatter |
|---|---|
| Only fire when I type `/name` | `disable-model-invocation: true` |
| Available but not listed in the picker | `user-invocable: false` |
| Run the skill in a [subagent](/guides/skills-vs-commands-vs-subagents) (separate context) | run it via a subagent-executing skill |

If a skill keeps firing when you don't want it to, the fix is almost always a tighter
`description` — not a config flag.

## A complete, useful example

A skill that reviews the current diff for one specific thing your team cares about:

```md
---
name: check-error-handling
description: Review the pending changes for swallowed errors and missing error handling. Use before committing, or when the user asks to check error handling.
---

Review only the pending diff (`git diff` and `git diff --staged`). For each changed file:

1. Flag any empty `catch {}` or `.catch(() => {})` — a swallowed error hides the failure.
2. Flag `async` calls with no `await` and no `.catch`.
3. Flag functions that can throw but are called without handling.
4. For each finding: quote the line, say what breaks, and suggest the fix.

Do NOT review anything outside the diff. Report "no issues found" if the diff is clean.
Be concrete — no generic advice.
```

Why this is a good skill and not a bloated one:

- **Scoped.** It reviews the *diff*, not the whole repo — so it is fast and its output is
  short.
- **Specific.** One concern (error handling), done well, beats a "review everything"
  mega-skill that does everything shallowly.
- **Trigger-friendly description.** "before committing" and "check error handling" are
  phrases you'd actually say.

## The mistakes that make a skill misfire

<Steps>
  <Step>
    ### A description that describes, instead of triggers

    `description: A skill for code quality.` never fires — Claude has nothing to match.
    Write when-to-use language: `Use before committing, or when the user asks to review
    the diff for bugs.`
  </Step>

  <Step>
    ### The mega-skill

    A 400-line SKILL.md that "handles the whole workflow" is worse than three focused
    skills. Claude follows a tight, single-purpose procedure reliably; a sprawling one it
    follows loosely. If your skill has "and also" in it, split it.
  </Step>

  <Step>
    ### Prose where steps belong

    "Consider the various factors and use good judgement" produces inconsistent results.
    Numbered, concrete steps produce consistent ones. Skills are procedures — write them
    like procedures.
  </Step>

  <Step>
    ### Doing a subagent's job in a skill

    If the skill says "search the whole codebase and read every match," all that output
    lands in your main context. That work belongs in a
    [subagent](/guides/skills-vs-commands-vs-subagents), which returns just the summary.
  </Step>
</Steps>

## Test it before you trust it

Invoke it with `/name` on a real case and read what it actually does. If it does the
wrong thing, the fix is in the body (tighten the steps) or the description (fix the
trigger). Iterate — a skill is prompt engineering, and prompts need a pass or two.

Writing one or two skills is an afternoon. Building and maintaining *dozens* that hand off
to each other is where it adds up — that is what a paid kit like
